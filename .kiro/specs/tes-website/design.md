# Design Document — TES Website

## Overview

The TenderEdge Solutions (TES) website is a single-page marketing and lead-generation site built
in the `apps/tes` workspace of the `pmg-hub` monorepo. It targets South African SMEs seeking
tender compliance services, drives WhatsApp enquiries as the primary conversion action, and
captures secondary leads via a server-side form backed by Astro Actions.

The site is a single scrolling document composed of ten section components assembled in
`pages/index.astro`. It is deployed to `tenderedgesolutions.co.za` on Vercel using Astro 6's
hybrid rendering mode — statically generated by default, with `prerender = false` on
`index.astro` to enable `getActionResult` server-side reads for the lead form.

### Design Philosophy

Dark Authority + Gold Precision. The aesthetic mirrors a specialist consultancy — serious,
precise, credible. Navy backgrounds (`#0b1929`) with gold accents (`#c9a227`) and off-white
body text (`#f0f4f8`). WhatsApp green (`#25D366`) is always preserved for CTA buttons —
muscle-memory recognition takes priority over colour harmony.

## Architecture

### Rendering Model

```
Vercel Edge Network
  └── Astro 6 (hybrid mode)
        ├── index.astro          prerender = false  → SSR on every request
        └── all other pages      prerender = true   → static at build time
```

`index.astro` must be SSR because `Astro.getActionResult()` requires a server context to read
the POST response from the `enquireLead` action. All other pages (if any are added later) remain
statically generated.

### Data Flow — Lead Capture

```
Visitor fills LeadForm
  → POST /  (native HTML form, method="POST", action={actions.enquireLead})
  → Astro Actions runtime intercepts POST
  → enquireLead handler runs server-side
        ├── Validates input (zod schema via astro:schema)
        ├── db.insert(leads) via @pmg/db  ──→  Neon PostgreSQL
        └── sendEmail(AdminNewLeadEmail)  ──→  Resend API
              (try/catch — email failure never blocks lead save)
  → Returns { success: true }
  → Astro re-renders index.astro with action result in context
  → LeadForm reads result via Astro.getActionResult()
  → Renders success state or inline field errors
```

### Monorepo Integration

| Concern | Package | Import |
|---|---|---|
| Database | `@pmg/db` | `import { db, leads } from '@pmg/db'` |
| Email | `@pmg/emails` | `import { sendEmail, AdminNewLeadEmail } from '@pmg/emails'` |
| Styles | `@pmg/tailwind-config` | `@import "@pmg/tailwind-config/base"` in globals.css |
| Analytics | `@vercel/analytics` | `import Analytics from '@vercel/analytics/astro'` |

### LeadForm — Action Integration

`LeadForm.astro` is the only component that interacts with Astro Actions. It reads the action
result in its frontmatter and conditionally renders the success state or the form with any
validation errors.

```astro
---
import { actions, isInputError } from 'astro:actions';

const result = Astro.getActionResult(actions.enquireLead);
const inputErrors = isInputError(result?.error) ? result.error.fields : {};
const submitted = result?.data?.success === true;
---
```

**Form markup pattern:**
```html
<form method="POST" action={actions.enquireLead}>
  <input type="text"   name="name"            required />
  <input type="text"   name="companyName" />
  <input type="tel"    name="phone"           required />
  <input type="email"  name="email" />
  <select              name="serviceInterest" required>...</select>
  <button type="submit">Send My Enquiry</button>
</form>
```

Field-level errors are rendered adjacent to each input using `inputErrors.fieldName?.[0]`.

## Data Models

### Astro Action Input Schema

Defined in `src/actions/index.ts` using `z` from `astro:schema`:

```ts
z.object({
  name:            z.string().min(1, 'Name is required'),
  phone:           z.string().min(7, 'Phone number is required'),
  email:           z.string().email().optional().or(z.literal('')),
  companyName:     z.string().optional().or(z.literal('')),
  serviceInterest: z.string().min(1, 'Please select a service'),
})
```

### Database Write — `leads` Table

The `leads` table is defined in `packages/db/src/schema/leads.ts`. The action inserts:

```ts
await db.insert(leads).values({
  name:            input.name,
  phone:           input.phone,
  email:           input.email || null,
  companyName:     input.companyName || null,   // stored in `message` field — see note
  serviceInterest: input.serviceInterest,
  source:          'tes',
  status:          'new',
});
```

> **Schema note:** The `leads` table does not have a `companyName` column. Company name should
> be stored in the `message` field as a prefixed string (e.g. `"Company: Basadipele Cleaning"`),
> or the schema should be extended with a migration to add a `company_name` column. The action
> implementation should handle this mapping. The `leads` table has a DB-level constraint
> requiring at least one of `email` or `phone` to be non-null — the form enforces `phone` as
> required, satisfying this constraint.

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of
a system — essentially, a formal statement about what the system should do. Properties serve as
the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Valid form submission persists a lead

*For any* valid form payload (non-empty name, phone ≥ 7 chars, valid or empty email, any
serviceInterest), calling the `enquireLead` handler SHALL insert exactly one row into the
`leads` table with `source = 'tes'`, `status = 'new'`, and field values matching the input.

**Validates: Requirements 13.3, 13.6**

### Property 2: Email failure does not prevent lead persistence

*For any* valid form payload where the email send throws or returns an error, the `enquireLead`
handler SHALL still insert the lead row and return `{ success: true }` — the email failure SHALL
NOT propagate to the caller.

**Validates: Requirements 13.5**

### Property 3: Invalid inputs are rejected before database write

*For any* form payload where a required field (name, phone, or serviceInterest) is empty or
phone is fewer than 7 characters, the `enquireLead` action SHALL return a validation error and
SHALL NOT insert any row into the `leads` table.

**Validates: Requirements 13.2, 12.10**

### Property 4: WhatsApp URL consistency

*For any* WhatsApp CTA rendered anywhere on the page, the `href` attribute SHALL equal
`https://wa.me/27745017094?text=Hi%2C+I'm+interested+in+your+tender+compliance+services.`
and the element SHALL carry `target="_blank"` and `rel="noopener noreferrer"`.

**Validates: Requirements 16.1, 16.2, 16.3**

### Property 5: Service card completeness

*For any* of the six service cards rendered by `ServicesSection`, the rendered HTML SHALL
contain the service name, a description, and a price string beginning with "R".

**Validates: Requirements 8.3, 8.4**

### Property 6: Pricing table row completeness

*For any* of the eleven individual service rows rendered by `PricingSection`, the rendered HTML
SHALL contain a service name, a price string, and a turnaround duration string.

**Validates: Requirements 10.5**

## Error Handling

### Form Validation Errors

Astro Actions uses zod for input validation. When validation fails, `isInputError(result.error)`
returns `true` and `result.error.fields` contains per-field error arrays. `LeadForm.astro` reads
these and renders inline error messages adjacent to each field.

### Action Handler Errors

If `db.insert` throws (e.g. DB connection failure), the action will propagate an unhandled error.
The form will not show a success state. A general error message is displayed:
`"Something went wrong. Please WhatsApp us directly."` — this is detected by checking
`result?.error && !isInputError(result.error)`.

### Email Send Errors

`sendEmail` from `@pmg/emails` never throws — it returns `{ data, error }`. The action wraps
the call in a try/catch as an additional safety net and logs any error to `console.error`. The
lead is already persisted at this point, so the error is non-fatal.

### DB Constraint — email/phone uniqueness

The `leads` table has unique indexes on `email` and `phone` (where not null). If a visitor
submits twice with the same phone number, the insert will throw a unique constraint violation.
The action should handle this gracefully — either by catching the constraint error and returning
a user-friendly message, or by using `onConflictDoNothing()` in the insert.

## Testing Strategy

### Unit Tests

Focus on the `enquireLead` action handler logic, isolated from the database and email service
using mocks.

- Valid payload → inserts lead with correct field mapping
- Empty name → validation error, no DB call
- Phone < 7 chars → validation error, no DB call
- Empty email string → stored as `null`
- Email send failure → lead still inserted, `{ success: true }` returned
- DB insert failure → error propagated, no success response

### Property-Based Tests

Use a property-based testing library (e.g. `fast-check` for TypeScript) to verify the
correctness properties defined above. Each test runs a minimum of 100 iterations.

**Property 1 — Lead persistence:**
Generate arbitrary valid payloads (non-empty name, phone ≥ 7 chars, optional email, any
serviceInterest string). For each, call the handler with a mocked DB and assert the insert was
called with `source: 'tes'`, `status: 'new'`, and matching field values.
Tag: `Feature: tes-website, Property 1: valid form submission persists a lead`

**Property 2 — Email failure resilience:**
Generate arbitrary valid payloads. Mock `sendEmail` to throw or return an error. Assert the
handler returns `{ success: true }` and the DB insert was still called.
Tag: `Feature: tes-website, Property 2: email failure does not prevent lead persistence`

**Property 3 — Invalid input rejection:**
Generate payloads with at least one invalid field (empty name, short phone, or empty
serviceInterest). Assert the action returns a validation error and the DB insert mock was never
called.
Tag: `Feature: tes-website, Property 3: invalid inputs are rejected before database write`

### Integration Tests

- Submit the form end-to-end in a local dev environment and verify a row appears in the `leads`
  table with `source = 'tes'`.
- Verify the admin notification email is received at `TES_ADMIN_EMAIL`.

### Visual / Smoke Tests

- All ten sections render without layout overflow at 320px, 768px, and 1280px viewport widths.
- Nav scroll behaviour: transparent at top, `bg-background/90` after 40px scroll.
- Mobile hamburger menu opens and closes correctly.
- Success state renders after valid form submission.
- Inline errors render after invalid form submission.
