# Requirements Document

## Introduction

This feature establishes a clean, scalable base schema structure for Drizzle ORM within the `packages/db` package. It introduces a domain-oriented schema layer split by business domain: divisions, clients, income, expenses, and leads. Each domain lives in its own file, and a barrel `index.ts` re-exports all tables and types for consumers.

Beyond schema definitions, this feature also covers the full Drizzle ORM + PostgreSQL package setup (connection, config, migrations), Drizzle relations, an idempotent seed script, and reusable typed query utilities.

## Glossary

- **Schema_Module**: A single TypeScript file under `packages/db/src/schema/` that defines Drizzle ORM table definitions and inferred types for one business domain.
- **Barrel_Index**: The `index.ts` file that re-exports all symbols from every Schema_Module.
- **Table**: A Drizzle ORM `pgTable` definition representing a database table.
- **Inferred_Type**: A TypeScript type derived from a Table using `$inferSelect` or `$inferInsert`.
- **Division**: An organisational unit used to group income and expense records.
- **Client**: A contact or business entity associated with income records.
- **Lead**: A prospective client or business opportunity tracked in the system.
- **Income_Record**: A financial record representing money received by a Division.
- **Expense_Record**: A financial record representing money spent by a Division.
- **DB_Package**: The `@pmg/db` workspace package located at `packages/db`.
- **DB_Connection**: The singleton Drizzle client instance exported from `packages/db/src/db.ts`.
- **Migration**: A SQL file generated by drizzle-kit that applies schema changes to the database.
- **Seed_Script**: An idempotent script that populates the database with baseline fixture data.
- **Query_Utility**: A reusable typed async function that encapsulates a common database query.

## Requirements

### Requirement 1: Package Setup

**User Story:** As a developer, I want the `packages/db` package fully configured for Drizzle ORM with PostgreSQL, so that I can run migrations and connect to the database without additional setup.

#### Acceptance Criteria

1. THE DB_Package SHALL include a `drizzle.config.ts` at the package root that points to the schema barrel index and a migrations output directory.
2. THE DB_Package SHALL include a `src/db.ts` file that creates and exports a single DB_Connection using `drizzle` with a `postgres` driver.
3. THE DB_Connection SHALL read the database URL from the `DATABASE_URL` environment variable.
4. IF the `DATABASE_URL` environment variable is not set at connection time, THEN THE DB_Package SHALL throw a descriptive error before attempting to connect.
5. THE DB_Package SHALL declare `drizzle-orm`, `drizzle-kit`, and `postgres` as dependencies in `package.json`.
6. THE DB_Package SHALL include a `.env.example` file at the package root containing the following template (with no real credentials):
```
# Required: PostgreSQL connection string
# Format: postgres://USER:PASSWORD@HOST:PORT/DATABASE
DATABASE_URL=postgres://user:password@localhost:5432/pmg_db
```
7. THE DB_Package README or inline comment in `db.ts` SHALL reference `.env.example` so that new developers know where to find the required environment variable format.

### Requirement 2: Divisions Schema Module

**User Story:** As a developer, I want a dedicated `divisions.ts` schema file, so that organisational unit data is isolated and easy to maintain.

#### Acceptance Criteria

1. THE Schema_Module SHALL be located at `packages/db/src/schema/divisions.ts`.
2. THE Schema_Module SHALL define a Table named `divisions` using Drizzle ORM's `pgTable`.
3. THE `divisions` Table SHALL include an `id` column of type `uuid` that is the primary key and defaults to a random UUID.
4. THE `divisions` Table SHALL include a `name` column of type `text` that is not null.
5. THE `divisions` Table SHALL include a `createdAt` column of type `timestamp({ withTimezone: true })` that defaults to the current time.
5a. THE `divisions` Table SHALL include an `updatedAt` column of type `timestamp({ withTimezone: true })` that is nullable, to be set by the application on update.
6. THE `divisions` Table SHALL define an index on the `name` column.
6a. THE `divisions` Table SHALL define a unique constraint on the `name` column.
7. THE Schema_Module SHALL export Inferred_Types for select and insert operations on the `divisions` Table.
8. IF a column constraint is violated at the database level, THEN THE DB_Package SHALL propagate the database error to the caller without swallowing it.

### Requirement 3: Income Schema Module

**User Story:** As a developer, I want a dedicated `income.ts` schema file, so that income records are isolated by domain and can evolve independently.

#### Acceptance Criteria

1. THE Schema_Module SHALL be located at `packages/db/src/schema/income.ts`.
2. THE Schema_Module SHALL define a Table named `income` using Drizzle ORM's `pgTable`.
3. THE `income` Table SHALL include an `id` column of type `uuid` that is the primary key and defaults to a random UUID.
4. THE `income` Table SHALL include a `date` column of type `date` that is not null.
4a. THE `date` column SHALL represent the transaction date - the date on which money was received - and SHALL NOT default to the current time.
5. THE `income` Table SHALL include a `divisionId` column of type `uuid` that is a foreign key referencing `divisions.id`, is not null, and defines `onDelete: "restrict"`.
6. THE `income` Table SHALL include a `clientId` column of type `uuid` that is a nullable foreign key referencing `clients.id` and defines `onDelete: "set null"`.
7. THE `income` Table SHALL include a `description` column of type `text` that is nullable.
8. THE `income` Table SHALL include an `amount` column of type `numeric(12,2)` that is not null.
9. THE `income` Table SHALL include a `createdAt` column of type `timestamp({ withTimezone: true })` that defaults to the current time.
9a. THE `income` Table SHALL include an `updatedAt` column of type `timestamp({ withTimezone: true })` that is nullable, to be set by the application on update.
10. THE `income` Table SHALL enforce a check constraint that `amount > 0`.
11. THE `income` Table SHALL define indexes on the `date`, `divisionId`, and `clientId` columns.
12. THE Schema_Module SHALL export Inferred_Types for select and insert operations on the `income` Table.
13. WHEN a `divisionId` references a non-existent Division, THE DB_Package SHALL propagate the referential integrity error to the caller.
14. WHEN a `clientId` references a non-existent Client, THE DB_Package SHALL propagate the referential integrity error to the caller.

### Requirement 3a: Clients Schema Module

**User Story:** As a developer, I want a dedicated `clients.ts` schema file, so that client contact data is isolated by domain and can be referenced from income records.

#### Acceptance Criteria

4. THE `clients` Table SHALL include a `name` column of type `text` that is not null, representing the contact or person name.
5. THE `clients` Table SHALL include a `businessName` column of type `text` that is nullable, representing an optional company name.
6. THE `clients` Table SHALL include an `email` column of type `text` that is nullable.
7. THE `clients` Table SHALL include a `phone` column of type `text` that is nullable.
8. THE `clients` Table SHALL include a `createdAt` column of type `timestamp({ withTimezone: true })` that defaults to the current time.
8a. THE `clients` Table SHALL include an `updatedAt` column of type `timestamp({ withTimezone: true })` that is nullable, to be set by the application on update.
9. THE `clients` Table SHALL define an index on the `name` column.
10. THE Schema_Module SHALL export Inferred_Types for select and insert operations on the `clients` Table.
11. THE `clients` Table SHOULD define a partial unique index on the `email` column where `email IS NOT NULL`, to prevent duplicate email addresses among clients who have one.

### Requirement 5: Leads Schema Module

**User Story:** As a developer, I want a dedicated `leads.ts` schema file, so that lead tracking data is isolated and can be extended without affecting other domains.

#### Acceptance Criteria

1. THE Schema_Module SHALL be located at `packages/db/src/schema/leads.ts`.
2. THE Schema_Module SHALL define a Table named `leads` using Drizzle ORM's `pgTable`.
3. THE `leads` Table SHALL include an `id` column of type `uuid` that is the primary key and defaults to a random UUID.
4. THE `leads` Table SHALL include a `name` column of type `text` that is nullable.
5. THE `leads` Table SHALL include an `email` column of type `text` that is nullable.
6. THE `leads` Table SHALL include a `phone` column of type `text` that is nullable.
7. THE `leads` Table SHALL include a `message` column of type `text` that is nullable.
8. THE `leads` Table SHALL include a `source` column of type `text` that is nullable.
9. THE `leads` Table SHALL include a `serviceInterest` column of type `text` that is nullable.
10. THE `leads` Table SHALL include a `status` column using a `pgEnum` with values `new`, `contacted`, `converted`, and `lost`, defaulting to `new`.
11. THE `leads` Table SHALL include a `createdAt` column of type `timestamp({ withTimezone: true })` that defaults to the current time.
11a. THE `leads` Table SHALL include an `updatedAt` column of type `timestamp({ withTimezone: true })` that is nullable, to be set by the application on update.
12. THE `leads` Table SHALL define indexes on the `status` and `createdAt` columns.
12a. THE `leads` Table SHALL define an index on the `email` column to support frequent lookups.
13. THE `leads` Table SHALL include a `divisionId` column of type `uuid` that is a nullable foreign key referencing `divisions.id` and defines `onDelete: "set null"`.
13a. THE `leads` Table SHALL define an index on the `divisionId` column.
14. THE Schema_Module SHALL export the `pgEnum` definition and Inferred_Types for select and insert operations on the `leads` Table.
15. THE `leads` Table SHALL enforce a CHECK constraint ensuring that at least one of `email` or `phone` is not null.
16. THE `leads` Table SHOULD store the `email` value normalized to lowercase before insertion to ensure consistent lookups regardless of input casing. This normalization SHALL be applied at the application layer before the insert. Note: The CHECK constraint ensures data capture quality but does not prevent duplicate leads. Deduplication logic is the responsibility of the application layer before insert.
17. THE `leads` Table SHOULD define a partial unique index on the `phone` column where `phone IS NOT NULL`, consistent with the partial unique index on email, to prevent duplicate phone numbers among leads who have one.
18. THE `leads` Table SHOULD define a partial unique index on the `email` column where `email IS NOT NULL`, to prevent duplicate email addresses among leads who have one. (Mirrors the pattern established in Req 3a criterion 11 for clients.)

### Requirement 6: Drizzle Relations

**User Story:** As a developer, I want Drizzle `relations()` helpers defined for all related tables, so that I can use Drizzle's relational query API without writing manual joins.

#### Acceptance Criteria

1. THE Schema_Module for `income.ts` SHALL define a `relations` export declaring that each Income_Record belongs to one Division via `divisionId`.
2. THE Schema_Module for `income.ts` SHALL define a `relations` export declaring that each Income_Record belongs to one Client (optional) via `clientId`.
3. THE Schema_Module for `expenses.ts` SHALL define a `relations` export declaring that each Expense_Record belongs to one Division via `divisionId`.
4. THE Schema_Module for `divisions.ts` SHALL define a `relations` export declaring that one Division has many Income_Records, many Expense_Records, and many Leads.
5. THE Schema_Module for `clients.ts` SHALL define a `relations` export declaring that one Client has many Income_Records.
6. THE Schema_Module for `leads.ts` SHALL define a `relations` export declaring that each Lead belongs to one Division (optional) via `divisionId`.
7. THE Barrel_Index SHALL re-export all `relations` definitions so they are available to the DB_Connection's schema configuration.

### Requirement 7: Barrel Index

**User Story:** As a developer, I want a single `index.ts` that re-exports all schema tables, types, and relations, so that consumers can import from one location without knowing the internal file layout.

#### Acceptance Criteria

1. THE Barrel_Index SHALL be located at `packages/db/src/schema/index.ts`.
2. THE Barrel_Index SHALL re-export all symbols from `divisions.ts`, `clients.ts`, `income.ts`, `expenses.ts`, and `leads.ts`.
3. WHEN a consumer imports from `@pmg/db`, THE DB_Package SHALL resolve all schema symbols through the Barrel_Index without requiring direct file imports.

### Requirement 8: Migrations

**User Story:** As a developer, I want drizzle-kit migration support configured and ready to run, so that I can apply schema changes to the database in a repeatable, ordered way.

#### Acceptance Criteria

1. THE DB_Package SHALL include a `drizzle.config.ts` that configures drizzle-kit with the schema path, migrations output directory, and database credentials.
2. THE DB_Package `package.json` SHALL include a `db:generate` script that runs `drizzle-kit generate`.
3. THE DB_Package `package.json` SHALL include a `db:migrate` script that runs `drizzle-kit migrate`.
4. WHEN migrations are generated, THE Migration files SHALL create the `divisions` table before the `income` and `expenses` tables to satisfy foreign key ordering.
5. IF a migration fails to apply, THEN THE DB_Package SHALL leave the database in its prior state without partial changes.

### Requirement 9: Seed Script

**User Story:** As a developer, I want an idempotent seed script, so that I can populate the database with baseline fixture data without creating duplicates on repeated runs.

#### Acceptance Criteria

1. THE DB_Package SHALL include a seed script at `packages/db/src/seed.ts`.
2. WHEN the seed script is run, THE Seed_Script SHALL insert exactly 2 Division records with names `TES` and `AWS` if they do not already exist.
3. WHEN the seed script is run, THE Seed_Script SHALL insert exactly 2-3 Client records if they do not already exist, before seeding income records.
4. WHEN the seed script is run, THE Seed_Script SHALL insert exactly 3 Income_Record fixtures associated with the seeded Divisions and referencing the seeded Clients if they do not already exist.
5. WHEN the seed script is run, THE Seed_Script SHALL insert exactly 3 Expense_Record fixtures associated with the seeded Divisions if they do not already exist.
6. WHEN the seed script is run, THE Seed_Script SHALL insert exactly 3 Lead fixtures if they do not already exist.
7. WHEN the seed script is run a second time without changes, THE Seed_Script SHALL produce no duplicate records and exit without error.
8. THE DB_Package `package.json` SHALL include a `db:seed` script that executes the seed script.
9. WHEN the seed script checks for existing records, THE Seed_Script SHALL use deterministic uniqueness checks (e.g., querying by `name` before inserting) rather than relying solely on `ON CONFLICT` clauses, to ensure idempotency across all supported database configurations.
10. THE Seed_Script SHALL execute all insert operations within a single database transaction so that a failure at any point results in a full rollback, leaving the database in its prior state.
11. APPLICATION LAYER NOTE: The `updatedAt` column is set by the application layer on update. This is a known tradeoff. Any database-level operation that bypasses the application (direct SQL fixes, migrations, external services) will leave `updatedAt` stale. Teams that require guaranteed accuracy SHOULD implement a PostgreSQL trigger as an alternative. Document this decision in a code comment above every `updatedAt` column definition.

### Requirement 10: Query Utilities

**User Story:** As a developer, I want reusable typed query functions for common reporting queries, so that I can retrieve aggregated financial and lead data without duplicating query logic across the codebase.

#### Acceptance Criteria

1. THE DB_Package SHALL include query utilities at `packages/db/src/queries.ts`.
2. THE Query_Utility `getTotalRevenue` SHALL return the sum of all `income.amount` values as a number.
3. THE Query_Utility `getTotalExpenses` SHALL return the sum of all `expenses.amount` values as a number.
4. THE Query_Utility `getRevenueByDivision` SHALL return an array of objects each containing a division name and the sum of income amounts for that division.
5. THE Query_Utility `getExpensesByDivision` SHALL return an array of objects each containing a division name and the sum of expense amounts for that division.
6. THE Query_Utility `getLeadsByStatus` SHALL return an array of objects each containing a status value and the count of leads with that status.
7. WHEN a Query_Utility is called and no matching records exist, THE Query_Utility SHALL return zero for sums (using `coalesce` at the SQL level to default `null` aggregate results to `0`) and an empty array for grouped results rather than throwing an error.
8. THE Query_Utility functions SHALL be fully typed with TypeScript return types inferred from the schema.
9. WHEN a Query_Utility reads a `numeric` column value returned by the driver, THE Query_Utility SHALL explicitly cast the value to a JavaScript `number` before returning it to the caller.
10. WHEN a grouped Query_Utility returns results (e.g., `getRevenueByDivision`, `getExpensesByDivision`, `getLeadsByStatus`), THE results SHOULD be ordered by total amount (or count for leads) in descending order by default.

### Requirement 11: Structural Consistency

**User Story:** As a developer, I want all schema files to follow the same conventions as existing files, so that the codebase remains uniform and easy to navigate.

#### Acceptance Criteria

1. THE Schema_Module SHALL use only named exports (no default exports).
2. THE Schema_Module SHALL prefix table names with a domain identifier (e.g., `divisions`, `income`, `expenses`, `leads`) consistent with existing naming conventions.
3. THE Schema_Module SHALL define Inferred_Types immediately after the Table definition they correspond to.
4. THE Schema_Module SHALL import Drizzle ORM helpers exclusively from `drizzle-orm/pg-core`.

### Requirement 12: Data Integrity Standards

**User Story:** As a developer, I want all schema modules and query utilities to follow consistent data integrity rules, so that the database is reliable, auditable, and safe to query across the entire codebase.

#### Acceptance Criteria

1. ALL monetary `amount` columns SHALL be stored as `numeric(12,2)` in the database. This applies to the `income` and `expenses` tables.
2. WHEN a Query_Utility reads a `numeric` column value returned by the database driver, THE Query_Utility SHALL explicitly cast the value to a JavaScript `number` before returning it to the caller.
3. WHEN a Query_Utility computes an aggregate (e.g., `sum`, `count`) and no matching records exist, THE Query_Utility SHALL use `coalesce` at the SQL level to default the result to `0` rather than returning `null`.
4. ALL timestamp columns in every Table SHALL use timezone-aware types (`timestamp({ withTimezone: true })` / `timestamptz`) to ensure consistent time representation across environments.
5. ALL Tables SHALL include a `createdAt` column of type `timestamp({ withTimezone: true })` defaulting to the current time, and SHOULD include an `updatedAt` column of type `timestamp({ withTimezone: true })` that is nullable and set by the application on update. APPLICATION LAYER NOTE: The `updatedAt` column is set by the application layer on update. This is a known tradeoff. Any database-level operation that bypasses the application (direct SQL fixes, migrations, external services) will leave `updatedAt` stale. Teams that require guaranteed accuracy SHOULD implement a PostgreSQL trigger as an alternative. Document this decision in a code comment above every `updatedAt` column definition.
6. ALL foreign key columns referencing `divisions.id` in the `income` and `expenses` tables SHALL define `onDelete: "restrict"` to prevent deletion of a Division that has associated financial records. Each such column definition SHALL include an inline code comment explaining this choice, e.g.: `// restrict: prevent division deletion while financial records exist`.
6a. THE `leads.divisionId` foreign key column referencing `divisions.id` SHALL define `onDelete: "set null"` so that deleting a Division nullifies the reference on associated leads rather than restricting or cascading. This differs intentionally from income/expenses because leads are soft-association data, not financial records. Each such column definition SHALL include an inline code comment explaining this choice, e.g.: `// set null: leads are soft-linked; division deletion should not block or cascade`.
7. ALL foreign key columns referencing `clients.id` SHALL define `onDelete: "set null"` so that deleting a Client nullifies the reference rather than cascading the deletion.
8. THE `divisions.name` column SHALL be unique across all rows in the `divisions` table.
9. THE `leads` Table SHALL enforce a CHECK constraint ensuring that at least one of `email` or `phone` is not null.
10. ALL foreign key columns and frequently queried columns (including `date`, `status`, `divisionId`, `clientId`, `email`) SHALL have database indexes defined to support efficient query execution.
11. ALL monetary values in Phase 0 are assumed to be in a single currency (ZAR - South African Rand). No currency column is required at this stage.
