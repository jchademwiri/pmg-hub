# Requirements Document

## Introduction

This feature establishes `packages/db` (`@pmg/db`) as the single shared database package for the PMG Hub monorepo. It uses Neon (serverless PostgreSQL) with Drizzle ORM and the neon-http driver. The scope is limited to setting up the package correctly: defining the full production schema, generating and applying migrations, exporting a clean public API, seeding initial data, and wiring the Turborepo pipeline. App integration (consuming `@pmg/db` in individual apps) is out of scope for this spec.

## Glossary

- **DB_Package**: The `packages/db` workspace package (`@pmg/db`)
- **Neon**: The serverless PostgreSQL provider used for all environments
- **Drizzle**: The ORM used for schema definition, migrations, and queries
- **Migration**: A SQL file generated by `drizzle-kit generate` and applied via `drizzle-kit push` or `drizzle-kit migrate`
- **Pooled_URL**: `DATABASE_URL` - the Neon connection pooler URL, used by app queries at runtime
- **Direct_URL**: `DATABASE_URL_UNPOOLED` - the Neon direct connection URL, used only for migrations
- **Lead**: A form submission from a public-facing Astro app (tes_leads, aws_messages, aws_bookings, pmg_leads)
- **Schema**: Drizzle table definitions in `packages/db/src/schema/`
- **Env_Validator**: The Zod-based environment validator in `packages/db/src/env.ts`

## Requirements

### Requirement 1: Environment Validation

**User Story:** As a developer, I want environment variables validated at startup, so that misconfigured deployments fail fast with a clear error rather than a cryptic runtime crash.

#### Acceptance Criteria

1. THE Env_Validator SHALL validate that `DATABASE_URL` is a valid URL string before the db client is constructed
2. THE Env_Validator SHALL validate that `DATABASE_URL_UNPOOLED` is a valid URL string before the db client is constructed
3. IF either variable is missing or malformed, THEN THE Env_Validator SHALL throw an error with a descriptive message listing the invalid fields
4. THE DB_Package SHALL use `DATABASE_URL` (pooled) for all runtime query connections
5. THE DB_Package SHALL use `DATABASE_URL_UNPOOLED` (direct) in `drizzle.config.ts` for migration commands only

### Requirement 2: Production Schema - Leads Tables

**User Story:** As a developer, I want a well-defined schema for all lead/enquiry tables, so that each app can insert structured data and the admin can query it consistently.

#### Acceptance Criteria

1. THE DB_Package SHALL define a `tes_leads` table with columns: `id` (uuid PK), `name` (text, not null), `email` (text, not null), `phone` (text, not null), `company` (text, nullable), `service_interest` (enum, nullable), `message` (text, not null), `newsletter_opt_in` (boolean, default false), `status` (enum, not null, default `new`), `is_read` (boolean, not null, default false), `notes` (text, nullable), `created_at` (timestamp, not null, defaultNow)
2. THE DB_Package SHALL define a `tes_service` enum with values: `bid_preparation`, `tender_tracking`, `compliance_docs`, `method_statements`, `pricing_boq`, `post_award`, `project_management`, `full_service`
3. THE DB_Package SHALL define a `tes_lead_status` enum with values: `new`, `contacted`, `converted`, `archived`
4. THE DB_Package SHALL define an `aws_messages` table with columns: `id` (uuid PK), `name` (text, not null), `email` (text, not null), `phone` (text, nullable), `subject` (text, nullable), `message` (text, not null), `newsletter_opt_in` (boolean, default false), `status` (enum, not null, default `new`), `is_read` (boolean, not null, default false), `notes` (text, nullable), `created_at` (timestamp, not null, defaultNow)
5. THE DB_Package SHALL define an `aws_bookings` table with columns: `id` (uuid PK), `name` (text, not null), `email` (text, not null), `phone` (text, not null), `package_name` (text, not null), `package_price` (integer, not null, in ZAR cents), `package_type` (enum, not null), `newsletter_opt_in` (boolean, default false), `status` (enum, not null, default `new`), `is_read` (boolean, not null, default false), `notes` (text, nullable), `created_at` (timestamp, not null, defaultNow)
6. THE DB_Package SHALL define an `aws_package_type` enum with values: `monthly`, `once_off`
7. THE DB_Package SHALL define an `aws_message_status` enum with values: `new`, `read`, `replied`, `archived`
8. THE DB_Package SHALL define an `aws_booking_status` enum with values: `new`, `contacted`, `active`, `completed`, `cancelled`
9. THE DB_Package SHALL define a `pmg_leads` table with columns: `id` (uuid PK), `name` (text, not null), `email` (text, not null), `phone` (text, nullable), `company` (text, nullable), `service_interest` (enum, not null, default `general`), `message` (text, nullable), `newsletter_opt_in` (boolean, not null, default false), `status` (enum, not null, default `new`), `is_read` (boolean, not null, default false), `notes` (text, nullable), `created_at` (timestamp, not null, defaultNow)
10. THE DB_Package SHALL define a `pmg_lead_service` enum with values: `tendering`, `web_dev`, `both`, `general`
11. THE DB_Package SHALL define a `pmg_lead_status` enum with values: `new`, `contacted`, `referred_tes`, `referred_aws`, `converted`, `archived`
12. THE DB_Package SHALL add a database index on the `status` column for each of the four lead tables
13. THE DB_Package SHALL add a database index on the `email` column for each of the four lead tables
14. THE DB_Package SHALL export TypeScript `$inferSelect` and `$inferInsert` types for each lead table

### Requirement 3: Production Schema - AWS Pricing Table

**User Story:** As an admin, I want pricing data stored in the database, so that it can be updated without a code deployment.

#### Acceptance Criteria

1. THE DB_Package SHALL define an `aws_pricing` table with columns: `id` (uuid PK), `name` (text, not null), `price` (integer, not null, in ZAR cents), `period` (text, nullable, e.g. `/month`), `upfront` (integer, nullable, setup fee in ZAR cents), `description` (text, not null), `features` (jsonb, not null, typed as `string[]`), `cta` (text, not null), `popular` (boolean, default false), `type` (`aws_package_type`, not null), `sort_order` (integer, default 0), `is_active` (boolean, default true)
2. THE DB_Package SHALL export `AwsPricing` and `NewAwsPricing` TypeScript types for the `aws_pricing` table

### Requirement 4: Remove Test Table

**User Story:** As a developer, I want the `connection_test` table removed from the production schema, so that the database only contains intentional production tables.

#### Acceptance Criteria

1. THE DB_Package SHALL NOT include the `connection_test` table in `src/schema/index.ts` after the production schema is in place
2. WHEN migrations are generated, THE DB_Package SHALL NOT produce a migration that creates the `connection_test` table

---

### Requirement 5: Migrations

**User Story:** As a developer, I want migrations generated and applied to Neon, so that the production database reflects the defined schema.

#### Acceptance Criteria

1. WHEN `bun db:generate` is run from the repo root, THE DB_Package SHALL generate SQL migration files in `packages/db/src/migrations/`
2. WHEN `bun db:migrate` is run from the repo root, THE DB_Package SHALL apply all pending migrations to the Neon database using the `DATABASE_URL_UNPOOLED` direct connection
3. THE DB_Package `drizzle.config.ts` SHALL use `DATABASE_URL_UNPOOLED` for the `dbCredentials.url` field
4. WHEN migrations are applied successfully, THE Neon database SHALL contain all tables defined in Requirement 2 and Requirement 3
5. THE root `package.json` SHALL expose `db:generate`, `db:migrate`, and `db:studio` scripts that delegate to `bun --filter @pmg/db`

---

### Requirement 6: DB Package Exports

**User Story:** As a developer consuming `@pmg/db`, I want a clean public API, so that I can import the db client and schema types without knowing the internal file structure.

#### Acceptance Criteria

1. THE DB_Package SHALL export the `db` Drizzle client instance from its main entry point (`"."`)
2. THE DB_Package SHALL export all schema table definitions and TypeScript types from its main entry point
3. THE DB_Package `package.json` exports field SHALL map `"."` to `"./src/index.ts"` for workspace consumers
4. WHEN a consuming app imports `from "@pmg/db"`, THE DB_Package SHALL resolve correctly via Bun workspace linking without requiring a build step

---

### Requirement 11: Seed Data - AWS Pricing

**User Story:** As a developer, I want initial pricing data seeded into the database, so that the AWS app has content to display without manual data entry.

#### Acceptance Criteria

1. THE DB_Package SHALL include a `src/seed.ts` script that inserts initial rows into `aws_pricing`
2. THE seed script SHALL insert 3 monthly packages: Starter (R299/mo), Growth (R599/mo, R1500 upfront, popular), Pro (R999/mo, R2500 upfront)
3. THE seed script SHALL insert 2 once-off packages: Landing Page (R2500), Business Website (R6500)
4. WHEN the seed script is run with `bun packages/db/src/seed.ts`, THE DB_Package SHALL insert the pricing rows without error
5. IF pricing rows already exist, THE seed script SHALL skip insertion rather than creating duplicates (idempotent)

---

### Requirement 12: Turborepo Pipeline

**User Story:** As a developer, I want the Turborepo pipeline aware of the db package, so that builds in consuming apps correctly depend on the db package being ready.

#### Acceptance Criteria

1. THE root `turbo.json` SHALL include a `db:generate` task definition so it can be run via `turbo run db:generate`
2. WHEN any app that depends on `@pmg/db` runs its `build` task, THE Turborepo pipeline SHALL ensure `@pmg/db` is resolved before the app build starts
